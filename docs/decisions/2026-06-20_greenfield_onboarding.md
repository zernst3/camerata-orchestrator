# 2026-06-20 Greenfield onboarding

## Context

Brownfield onboarding (scan -> propose -> audit -> arm) handles existing repos. The
USER_GUIDE describes a parallel path: greenfield onboarding scaffolds a BRAND NEW
local git repo with governance files baked in from commit zero. The Onboard view
already had a "Greenfield" path selector and step descriptions, but no implementation
behind the "Scaffold repo" button.

## Decisions

### 1. Scaffold function lives in `onboard.rs` (not a new module)

`scaffold_greenfield_blocking` is added to `crates/server/src/onboard.rs` alongside
the brownfield scan/audit logic. Rationale: greenfield is the zero-existing-code case
of the same onboarding concern. Keeping them co-located avoids a module boundary where
there is no semantic separation — both paths are answering the same question ("get this
repo under governance").

### 2. Emit reuses `arm_files_for_repo` exactly — no duplication

The governance files written by greenfield scaffold are generated by the same
`crate::arm::arm_files_for_repo` call that the brownfield apply path uses. This is not
optional — it is a correctness invariant. If AGENTS.md / CONVENTIONS.md / rules.json /
ci-checks.json / the CI workflow ever change shape, brownfield and greenfield are
identical. Duplication here would mean one path silently drifts.

### 3. `scaffold_greenfield_blocking` is synchronous (via `spawn_blocking`)

Filesystem + git operations are blocking. The function is called via
`tokio::task::spawn_blocking` in the handler so the async runtime is not blocked. The
blocking API is cleaner than async-file + async-process for a sequence that is
inherently serial.

### 4. Refuse to clobber an existing directory

The function bails immediately if `dest` already exists. Greenfield = a brand-new repo.
Clobbering an existing directory would be data loss with no recovery path. The error
message is actionable: the UI shows it via the `gf-result-err` panel.

### 5. Git identity fallback for clean environments

In CI / fresh machines with no global git config, `git commit` fails without a
user.email and user.name. The scaffold sets per-repo config values (`camerata@example.com`
/ `Camerata`) before committing. These are local overrides only (written to `.git/config`,
not `--global`).

### 6. Handler marks the repo onboarded in the active project

After a successful scaffold the handler calls `projects.mark_onboarded` on the active
project (creating one from the repo name if none is active). This makes the greenfield
path equivalent to the brownfield complete step: the repo appears in the project's repos
list and the onboarded badge lights up.

### 7. Handler records the local path override

`settings.set_repo_path` stores the scaffolded directory as the repo's machine-local
path. Subsequent scan / apply / audit operations resolve this repo to its directory
without needing a workspace root.

### 8. UI: greenfield form is a dedicated `GreenfieldForm` component

The brownfield and greenfield paths share the path-chooser and the step-list (shown
before action), but diverge completely in their action forms. A separate `GreenfieldForm`
component keeps the signal surface clean: the parent `OnboardView` holds the greenfield
signals and passes them as `Signal<T>` props so they survive path switches.

### 9. Starter ruleset picker: full corpus, recommended pre-ticked

The greenfield picker shows the full corpus rule library. Rules with `recommended: true`
are pre-selected (same signal the brownfield proposed-rules table uses). The architect
can untick any recommended rule and tick any available one. This matches the USER_GUIDE
description: "Start from the corpus defaults for your stack; edit and approve."

The picker is a compact checklist (not the full chorale table) because greenfield has no
scan results or finding counts to display — the architect is picking prospectively, not
triaging findings.

### 10. Directive resolution for the scaffold request

When the greenfield form sends rules to the handler it resolves each rule's `directive`
from `ProposedRuleView.default_option -> options[*].directive`, falling back to
`title`. This mirrors the brownfield apply path's directive resolution
(`resolve_directive` in `ScanResults`). The server receives resolved directives so the
`arm_files_for_repo` emit produces correct output.

### 11. The greenfield path does NOT push to GitHub

The scaffold is intentionally LOCAL-ONLY. The architect creates the remote (e.g. on
GitHub) and wires it up themselves. This matches the local-first principle and avoids
requiring a GitHub token for a new repo that may not yet exist on GitHub.
