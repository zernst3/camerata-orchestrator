# Ruby on Rails, Python Django, and Python Flask framework corpora

Status: accepted (2026-06-20)

## Context

The Camerata rule corpus previously covered universal patterns, JavaScript (Next.js, Express, React,
Redux, TypeScript), Rust (domain, SeaORM, Dioxus), Java, Go, C#, SQL, CI/CD, IaC, and other
cross-cutting layers. Two high-adoption backend ecosystems with distinct idiom sets were absent:
Ruby on Rails and Python's two dominant web frameworks (Django and Flask). These are common targets
for Camerata's brownfield onboarding flow and project governance scans.

## Decision: add three framework subdir corpora

Three new subdirectory corpora are added under `crates/rules/principles/`:

- `ruby/rails/` with domain `ruby:rails`
- `python/django/` with domain `python:django`
- `python/flask/` with domain `python:flask`

Each follows the established convention: framework rules live in a `<lang>/<framework>/` subdir,
carry `domain = "<lang>:<framework>"`, and `layer = "framework"`.

## Coverage rationale

Each rule targets a class of real, commonly-found violations in the respective ecosystem:

- **SQL injection / parameterization**: all three frameworks get a mechanical-enforcement rule
  covering their specific raw-SQL escape hatches (Rails `.where` string interpolation, Django
  `.extra()` / `.raw()` with f-strings, Flask + SQLAlchemy `text()` misuse).
- **N+1 queries**: Rails (`includes`/`eager_load`) and Django (`select_related`/`prefetch_related`)
  each get a structured rule; Flask is ORM-agnostic enough that this is deferred to the SQLAlchemy
  layer.
- **Layering / thin controllers**: all three frameworks get a structured rule reinforcing the
  same separation-of-concerns principle present in Go, Express, Java, and C# rules.
- **Input validation**: each framework gets a rule naming its idiomatic validation primitive
  (Rails Strong Parameters, Django Forms/Serializers, Flask marshmallow/pydantic).
- **CSRF + auth**: Rails and Django get framework-specific CSRF rules; Flask gets an explicit
  auth-on-routes rule since Flask has no built-in CSRF.
- **Secrets / config from env**: all three frameworks get a mechanical-enforcement rule.
- **Rails-specific**: scopes (reuse over ad-hoc where chains), model validations (all entry points,
  not just HTTP), and background jobs (Sidekiq / GoodJob / Solid Queue pattern).
- **Django-specific**: migrations checked in (deterministic schema evolution).
- **Flask-specific**: app factory + blueprints (testing isolation), error handlers (consistent
  JSON error envelopes).
