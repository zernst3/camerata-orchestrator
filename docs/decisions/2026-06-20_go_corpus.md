# 2026-06-20: Go Rule Corpus

**Status:** IMPLEMENTED

## What

Added a complete Go rule corpus (10 rules) to camerata under `crates/rules/principles/go/` that mirrors the Python and Rust corpus structure and format. The corpus covers:

1. **Security / Error Handling**
   - GO-ERRORS-MUST-BE-CHECKED-1 (mechanical): Every error return must be explicitly checked
   - GO-SQL-PARAMETERIZED-1 (mechanical): SQL queries use bound parameters, never string concatenation
   - GO-ERROR-WRAPPING-WITH-FMT-W-1 (prose): Errors wrapped with %w to preserve chains

2. **Concurrency / Resource Management**
   - GO-CONTEXT-PROPAGATION-1 (structured): Context flows through call stacks and goroutines
   - GO-GOROUTINE-LEAKS-DEFER-CLEANUP-1 (structured): Resources cleaned via defer; goroutines never leak
   - GO-NO-GOROUTINE-GLOBALS-1 (structured): Mutable state guarded by locks or not global

3. **Architecture / Layering**
   - GO-HANDLER-SERVICE-REPOSITORY-1 (structured): HTTP handlers → services → repositories
   - GO-PACKAGE-BOUNDARIES-CLEAR-1 (mechanical): No circular imports; internal/ packages enforced
   - GO-SMALL-INTERFACES-1 (prose): Interfaces small and cohesive

4. **Idiom / API Design**
   - GO-ACCEPT-INTERFACES-RETURN-STRUCTS-1 (prose): Accept interfaces, return structs; interfaces at consumer
   - GO-LOGGING-STRUCTURED-1 (prose): Structured logging (slog/zap), not Printf

## Why

The camerata corpus is domain-specific (language + framework + paradigm). Go is:
- A statically-typed compiled language with built-in concurrency primitives (goroutines, channels, defer)
- Designed for fast, reliable backends with a strong idiom-based culture (error wrapping, interface segregation, small packages)
- Increasingly used in cloud infrastructure and microservices

The corpus fills a gap for Go projects that adopt camerata, providing:
- **Stack detection**: `.go` files already detected; added "go" case to `domain_to_glob` in lib.rs
- **Deterministic + AI-auditable rules**: Mix of mechanical (errcheck, circular imports), structured (layering guidance), and prose (idiom) rules
- **Production-ready framing**: Each rule has decision questions, defaults, and 1-2 alternatives

## How

### Corpus TOML Files

Created 10 rules under `crates/rules/principles/go/`:
- `go-errors-must-be-checked-1.toml` — error handling discipline
- `go-sql-parameterized-1.toml` — SQL injection prevention
- `go-error-wrapping-with-fmt-w-1.toml` — error chain preservation
- `go-context-propagation-1.toml` — request lifetime management
- `go-goroutine-leaks-defer-cleanup-1.toml` — resource cleanup and goroutine safety
- `go-no-goroutine-globals-1.toml` — concurrency safety
- `go-handler-service-repository-1.toml` — layered architecture
- `go-package-boundaries-clear-1.toml` — module structure
- `go-small-interfaces-1.toml` — interface design
- `go-accept-interfaces-return-structs-1.toml` — API design idiom
- `go-logging-structured-1.toml` — observability

Each TOML follows the camerata corpus schema:
- `id` (e.g., GO-ERRORS-MUST-BE-CHECKED-1)
- `title`, `tag` (always "stack"), `domain` (always "go"), `layer`, `enforcement` (prose/structured/mechanical)
- `qualifies` (one-paragraph scope / detection method)
- `[decision]` section with `question`, `default`, `why`
- `[[option]]` blocks with `id`, `label`, `directive`, `why`
- All rules ship a `default = true` adoption (option is selected)

## Usage

Go projects adopting camerata can now:
```rust
let go_backend_role = role_from_corpus(
    path,
    "Go Backend",
    &["go", "sql", "universal"],  // domain filter
    &[],
).await?;
```

The role will include:
- All 10 Go-specific rules
- Universal rules (domain = "*")
- Allowed paths: `**/*.go`, `**/*.sql`, `**` (catch-all)

## Alternatives Considered

1. **Single large Go interface** — Rejected: Small interfaces are the Go idiom; they compose better and are easier to mock.
2. **Parameterized queries vs. ORM-only** — Unified under "bind-parameters-or-query-builder" because both satisfy the safety constraint; ORM query builders generate parameterized SQL internally.
3. **Enforcement tiers** — Chose mix of mechanical (errors, SQL, packages), structured (context, layering, concurrency), and prose (idiom, API design) to reflect Go's balance of compile-time checks and runtime culture.
