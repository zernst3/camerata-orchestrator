# 2026-06-23: Scan hygiene — gitignore-aware walking and self-referential suppression

## Context

Two classes of false positives emerged from the brownfield local scan:

1. **Build artefacts and secrets files that are already gitignored.** The original DFS
   walk used a static noise denylist (`NOISE_DIRS` / `NOISE_FILES` / `NOISE_SUFFIXES`).
   Any gitignored file that was not already on that list was scanned even though the
   developer had already told git (and every git-aware tool) to ignore it. This produced
   spurious findings on generated files, gitignored build outputs, and developer-local
   secret files that were intentionally excluded from the repo.

2. **Rule descriptions quoted verbatim inside Camerata-emitted governance files.**
   `CONVENTIONS.md` and `AGENTS.md` are generated by Camerata and carry the rule
   directives (e.g. "Do not set `verify=False` in TLS configuration") as human-readable
   text. The content scanner, which runs over every file in the repo, fires on those
   description strings and produces a finding that is not a real violation. The same
   applies to corpus TOML files (the rule principals themselves) and to `.camerata/`
   config files.

## Decision 1: Gitignore-aware file walking

The local scan now uses the `ignore` crate (ripgrep's gitignore engine) when the
scanned directory is a git repo (`.git` exists). The walker respects:

- `.gitignore` at the repo root and in any subdirectory
- `.git/info/exclude` (the per-repo non-committed exclude list)
- The user's global gitignore (`core.excludesFile`)

The noise denylist (`is_noise_path`) is applied on top as belt-and-suspenders for repos
whose build artefacts are not yet gitignored.

When the scanned directory is NOT a git repo (`.git` absent), the function falls back to
the original iterative DFS noise-denylist walk. This preserves the original behaviour for
non-repo directories and avoids breaking callers that use a temporary scratch directory
during tests.

### Semantics that matter

- A gitignored `.env` **is skipped** — the developer explicitly opted it out.
- A committed `.env` that is NOT in `.gitignore` **is still scanned** — it is tracked
  source, and if it contains a real credential that credential is genuinely in the repo.
  This is the correct behaviour: `ignore` honours the gitignore file, not the git index,
  so the decision to scan or skip is always "is this file gitignored?" not "is it staged?"
- `hidden(false)` is set so dotfiles like `.env`, `.camerata/`, and `.env.local` are
  visited by the walker. Without this the `ignore` crate's default behaviour would skip
  them silently.

## Decision 2: Self-referential finding suppression

A new suppression tier, `suppressed-self-reference`, is added alongside the existing
`suppressed-inline` and `suppressed-baseline` statuses.

A finding is self-referential when BOTH conditions hold:

1. The file is a **governance or corpus artifact** — defined as any of:
   - A file whose content contains the `<!-- Generated by Camerata` header (the standard
     header Camerata writes into every AGENTS.md and CONVENTIONS.md it emits).
   - A file whose path is under `.camerata/` (project-level governance config).
   - A `.toml` file whose path contains a `principles/` segment (a rule corpus TOML).

2. The matched snippet **appears in corpus descriptive text** — collected from rule
   titles, summaries, `decision_why` fields, and option `directive`/`why` strings.
   Matching is case-insensitive substring. An empty snippet never matches.

When both conditions hold, the finding's `status` is set to
`suppressed-self-reference` and the `detail` field is annotated.

### Safety properties

- **Governance-file scoped.** A real `verify=False` in `app/config.py` stays `active`
  even though the SEC-NO-DISABLED-TLS-1 rule description contains that string. Condition
  1 fails for ordinary source files.
- **Real secrets pasted into governance files still flag.** A developer who pastes a real
  token into `CONVENTIONS.md` produces a finding whose snippet (`ghp_…`) is not in any
  rule description. Condition 2 fails, so the finding stays `active`.
- **The gateway source (`crates/gateway/src/lib.rs`) is outside scope.** It is
  application source and handled by repo-level scan exclusion or `camerata:allow`, not
  by this mechanism.
- **Already-classified findings are never downgraded.** `suppress_self_referential`
  only touches findings whose `status == "active"`. Inline and baseline suppressions
  are preserved.

## UI change

The status badge in the cockpit findings table now maps `suppressed-self-reference` to
the label `"self-ref"` with a gray `BadgeVariant` ("Self-referential"). This sits
alongside the existing `"Baseline debt"` (gray) and `"Waived"` (yellow) labels.
`suppressed-self-reference` is correctly excluded from the enforced-count computation
at line 9962 of `cockpit.rs` (the filter is `status == "active"`), so no additional
change is needed there.

- `crates/server/Cargo.toml`: `ignore = { workspace = true }` added to `[dependencies]`.
